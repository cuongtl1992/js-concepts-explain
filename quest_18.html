The Performance Problem with Sequential Async Tasks

When using async/await in a loop as shown in the original code:

async function processItems(items) { for (const item of items) { await doAsyncTask(item); // Wait for each task to complete before starting the next } console.log("All tasks completed"); }

This creates a sequential execution pattern where:

  • Each async task must complete before the next one starts
  • Total execution time = sum of all individual task times
  • If you have 100 tasks that each take 100ms, the total time will be 10,000ms (10 seconds)

This sequential approach becomes increasingly inefficient as:

  • The number of items to process increases
  • Each individual task involves waiting (like network requests, file I/O, etc.)

Why This Is Inefficient

Most asynchronous operations (like HTTP requests, database queries, etc.) involve a lot of waiting time where the JavaScript engine is idle. During these wait periods, the engine could be processing other tasks in parallel.

In the sequential approach, we're essentially:

  • Start Task 1 → Wait for completion
  • Start Task 2 → Wait for completion
  • And so on...

This is like waiting in line at a single checkout counter, when multiple counters could be open.

Solution: Parallel Execution with Promise.all()

Instead of waiting for each task to complete before starting the next one, we can start all tasks at once and wait for all of them to complete:

async function processItemsInParallel(items) { // Create an array of promises by mapping each item to its task const promises = items.map(item => doAsyncTask(item)); // Wait for all promises to resolve await Promise.all(promises); console.log("All tasks completed"); }

Promise.all() takes an array of promises and returns a new promise that resolves when all input promises have resolved.

Benefits of Parallel Execution

  • Significantly reduced total execution time - Total time ≈ duration of the longest task
  • Better resource utilization - JavaScript engine and browser can manage multiple operations simultaneously
  • Improved user experience - Faster overall processing time

When to Use Promise.all vs Sequential Processing

Use Promise.all When: Use Sequential Processing When:
Tasks are independent of each other Each task depends on the result of the previous task
You want to minimize total execution time You need to control resource usage (e.g., limit concurrent requests)
You're dealing with a large number of items Tasks need to be processed in a specific order
Tasks involve waiting (I/O, network, etc.) You need to handle errors before proceeding to the next task

Handling Errors with Promise.all

One important thing to note: if any of the promises in the array passed to Promise.all() rejects, the entire operation will reject immediately:

async function processItemsWithErrorHandling(items) { try { const promises = items.map(item => doAsyncTask(item)); await Promise.all(promises); console.log("All tasks completed successfully"); } catch (error) { console.error("One of the tasks failed:", error); // Handle the error appropriately } }

Alternative: Promise.allSettled()

If you want to wait for all promises to complete regardless of whether they fulfill or reject, you can use Promise.allSettled():

async function processItemsAllSettled(items) { const promises = items.map(item => doAsyncTask(item)); const results = await Promise.allSettled(promises); // Process results const successful = results.filter(r => r.status === 'fulfilled'); const failed = results.filter(r => r.status === 'rejected'); console.log(`${successful.length} tasks succeeded, ${failed.length} failed`); }
